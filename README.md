## Setup Instructions

### 1. Install HubSpot CLI

    npm install -g @hubspot/cli

### 2. Configure Environment Variables

Copy the example file:

    cp .env.example .env.test

Edit .env.test and add your credentials:

    HUBSPOT_PORTAL_ID=your-test-portal-id
    HUBSPOT_ACCESS_TOKEN=your-test-access-token
    ENVIRONMENT=test
    PRODUCTION_PORTAL_ID=your-prod-portal-id

### 3. Authenticate HubSpot CLI

    hs auth

## Usage

All scripts load env via `dotenv`, defaulting to `.env`. To target the test or prod portal,
set `DOTENV_CONFIG_PATH` and use `node -r dotenv/config`:

    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/<script>

### Build Configs from CSVs (Run First, after CSV edits)

Converts `data_files/*.csv` → `config/*.json` (the API payloads). Idempotent.

    node scripts/build-configs.js

### Audit Target Portal (Recommended Before Create or Delete)

Read-only. Lists existing groups + `pd_*` properties and diffs them against `config/`.
Saves a snapshot to `extracts/portal-<id>-<ts>.json`.

    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/extract-portal-state.js

### Create Property Groups (Run Before Properties)

    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/create-groups.js

### Create Properties

    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/create-properties.js

### Cleanup Test Portal (Test Only)

Both delete scripts hard-code the test portal ID (`51225486`) and refuse to run against any
other portal — there is no override flag. To remove properties from any other portal
(including production), use the HubSpot UI manually.

Only deletes properties with the `pd_` prefix. Run the property delete first; the group
delete will fail if a group still contains properties.

    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/delete-properties.js
    DOTENV_CONFIG_PATH=.env.test node -r dotenv/config scripts/delete-groups.js

## Security & Compliance

This project follows Maverick Payments security requirements:

**6.3 API Security:**
- Uses Private Apps only (not OAuth)
- Tokens scoped to minimum required permissions
- Credentials stored in .env files (excluded from Git)
- Delete scripts hard-code the test portal ID as an allowlist; no override exists
- Delete operations limited to `pd_*` prefix properties
- `extract-portal-state.js` is read-only and supports pre-cutover audits against any portal

**Required Scopes (Private App):**

    crm.schemas.contacts.read
    crm.schemas.contacts.write
    crm.schemas.companies.read
    crm.schemas.companies.write
    crm.schemas.deals.read
    crm.schemas.deals.write
    crm.schemas.leads.write

The `read` scopes are required for `extract-portal-state.js` and `delete-properties.js`
(which list properties before deleting). Leads has no schema-level read scope; the leads
write scope grants sufficient access to list lead properties.

## Workflow

1. **Test Portal**: Run scripts in test environment
2. **Validation**: Verify all properties created correctly
3. **Security Review**: Submit to Maverick security team
4. **Approval**: Get sign-off from security team
5. **Production**: Deploy to production portal

## Safety Features

- Environment variable validation before execution
- Delete scripts use a hard-coded test portal allowlist (`51225486`) with no override flag —
  refuse to run against any other portal
- Delete operations only target properties with `pd_` prefix
- `extract-portal-state.js` provides read-only pre-cutover audit (NEW / MATCH / CONFLICT
  diff against config) so the prod cutover can be verified before any write
- Separate branches for test and production
